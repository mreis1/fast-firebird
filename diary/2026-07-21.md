# 2026-07-21 — Session 45: docs site (GitHub Pages) + FB6 in the demo dashboard

## FB6 in the demo dashboard

`fb6` added to the demo's built-in matrix (`apps/demo/server/servers.ts`,
port 30507, version 6) and the feature explorer extended with a Firebird 6
section (`apps/demo/server/features.ts`, `since` widened to `3|4|5|6`, new
`.badge.since6` style).

Candidate FB6 features were qualified empirically before inclusion — a probe
ran each SQL against the live fb5 (5.0.1) and fb6 (6.0.0.2078) containers,
and only FB5-fail/FB6-pass features shipped:

- **SQL schemas** — `create schema if not exists` + qualified
  `ff_feat_schema.parts` table + `current_schema`. Two traps:
  `CURRENT_SCHEMA` is a reserved word (can't alias a column as itself —
  aliased `cur_schema`), and `drop schema` fails on a non-empty schema, so
  the demo setup relies on `if not exists` + `recreate table` for
  re-runnability.
- **`CAST … FORMAT`** — format masks work both directions; the AM/PM token
  is **`P.M.`**, not `AM` ("Cannot recognize A part of date format").
- **`GREATEST`/`LEAST`/`BTRIM`** — all three FB6-only (BTRIM with a custom
  trim character).
- **`ANY_VALUE`** — aggregate; demoed with a GROUP BY bucket query.

Rejected: row-value constructors `(values (1,'a'),(2,'b'))` — fails on FB6
too; `RDB$GET_CONTEXT('SYSTEM','SCHEMA_SEARCH_PATH')` — not a valid context
variable on the snapshot.

Verified end-to-end through the running demo API: fb6 connects (protocol 20,
Arc4, engine 6.0.0), all four features return rows via `/try-feature`, and
all three SQL-runner lanes (core, drizzle, node-firebird2-ext compat) execute
against FB6.

## Docs site (`docs/`, VitePress → GitHub Pages)

Phase 0 item 1 (+ item 2's migration guide) of plans/reference-strategy.md.

- **Stack**: VitePress 1.6.4 as a workspace package (`@fast-firebird/docs`,
  `docs` added to pnpm-workspace.yaml), TypeDoc 0.28 +
  typedoc-plugin-markdown + typedoc-vitepress-theme generating the API
  reference from `packages/core/src/index.ts` into `docs/api/` (gitignored —
  build artifact; `pnpm --filter @fast-firebird/docs build` = typedoc +
  vitepress build). The VitePress config reads `api/typedoc-sidebar.json`
  with an fs fallback to `[]` so `vitepress dev` works before the first
  TypeDoc run.
- **Content**: 19 guide pages — the README split per topic (queries,
  transactions, batch, streaming, blobs, types, prepared statements, pool,
  scripts, events, services, CHARSET NONE, FB6, security, performance,
  Drizzle) plus a landing hero and a new **public node-firebird migration
  guide** (side-by-side code groups for connect/query/sequentially/
  transaction/pool/blobs, an isolation mapping table, a feature-by-feature
  map, and a "watch out for" list: Node ≥ 22, wireCrypt default, UTF8
  default, tx-scoped lazy blob handles, queryOne fetches all).
- **Accuracy check that paid off**: the first draft of the types page said
  NUMERIC/DECIMAL decodes to string. Probed against FB5: scaled
  NUMERIC/DECIMAL decodes to JS **number** (`applyScale`, documented lossy
  beyond 2^53 — `12345678901234567.89` → `12345678901234568`); scale-0
  BIGINT follows `int64As: 'auto'` (number when safe, else bigint); DECFLOAT
  is the exact-string type. The published table now matches the code, points
  exact-money at DECFLOAT or `cast(… as varchar)`, and documents `int64As`.
- **Deploy**: `.github/workflows/docs.yml` — push-to-main (paths: docs/,
  core/src/, the workflow) + manual dispatch; pnpm install with the same
  references/ stub as ci.yml, build, `upload-pages-artifact` from
  `docs/.vitepress/dist`, `deploy-pages`. Site base `/fast-firebird/` →
  https://mreis1.github.io/fast-firebird/. One-time repo setting needed:
  Pages → Source → **GitHub Actions**.
- Verified: full build green locally; rendered site (home, migration guide
  code-group tabs, API reference tables/sidebar) inspected in a browser via
  `vitepress preview`.
