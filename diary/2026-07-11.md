# 2026-07-11

## Session 21 — the 2.8 MB/s mystery SOLVED: Tcp1323Opts=0 on the server VM

Conclusion of the remote-fetch investigation (sessions 17–20). Full causal
chain, for posterity:

### The measured facts
- Firebird fetch: flat ~2.8 MB/s per connection (any row count); 8 parallel
  connections → 10.4 MB/s aggregate. Insert direction ~20 MB/s.
- Driver exonerated: blob read pipeline verified symmetric with writes at
  60 ms simulated RTT (~4 RTs/blob both ways).
- Topology clarified along the way: there is NO VPN — the "Windows 10 Pro
  box" IS the Firebird server, a virtio-NIC'd VM (machine pc-i440fx-5.1)
  on an OVH-hosted Proxmox with a 10 Gbit host link. Datacenter bandwidth
  limits: none. NIC rate limit: none.

### The smoking gun
`iperf3 -c <server-ip>` from the Mac: 25.8 Mbit/s with **Cwnd pinned at
exactly 64.0 KBytes for the whole run and 0 retransmits**. Zero loss + a
window that never grows = hard clamp, not congestion. 64 KB ÷ ~20 ms RTT
= 3.2 MB/s — the measured number, both directions, Firebird or iperf alike.

### Root cause
`HKLM\SYSTEM\CurrentControlSet\Services\Tcpip\Parameters\Tcp1323Opts = 0x0`
on the Windows server VM — an XP-era "tuning" relic that disables RFC 1323
**window scaling** (and timestamps) machine-wide. Without wscale the TCP
window field is 16-bit → every connection caps at 65,535 bytes in flight,
regardless of `netsh autotuninglevel=normal` (which showed healthy — the
netsh globals are moot when 1323 options are registry-disabled). This also
explains the earlier netsh line "Horodatages RFC 1323: disabled".

Diagnostic breadcrumbs that pointed here: per-flow cap identical in BOTH
directions (send and receive) → not autotuning (receive-only) and not the
app → the scaling OPTION itself missing from the handshake.

### Fix
```
reg add HKLM\SYSTEM\CurrentControlSet\Services\Tcpip\Parameters /v Tcp1323Opts /t REG_DWORD /d 1 /f
```
(1 = window scaling on, timestamps off = modern default; deleting the value
is equivalent) + **reboot**. Verify: iperf3 Cwnd grows past 64 K; demo
custom bench fetch conns=1 should jump from 2.8 MB/s to the OVH plan's
public-bandwidth ceiling.

### Ruled out en route (kept for future triage value)
- Proxmox: datacenter bandwidth limits, NIC model (virtio ✓), per-NIC rate=,
  host link (10 Gbit ✓).
- VPN: none exists (three checks: ncpa.cpl, Get-VpnConnection, ipconfig).
- Windows autotuning + heuristics: normal / disabled ✓.
- OVH shaping: not the per-flow cap; the ~10.4 MB/s aggregate may still be
  the plan's public-bandwidth profile — re-measure aggregate after the fix.
- Driver/app layer: pipelined, symmetric, and `pool.map` parallel fetch
  (demo "fetch conns") remains a useful lever on any window-limited path.

## Session 22 — fix confirmed: 25 Mbit → 356 Mbit (14×)

- Applied `Tcp1323Opts = 1` + reboot on the server VM.
- **Gotcha worth remembering**: the first post-reboot `iperf3` still showed
  25 Mbit / Cwnd 64K — because Windows iperf3 builds (cygwin) set a 64 KB
  socket buffer by default, which disables autotuning ON THE TEST SOCKET and
  perfectly mimics the original symptom. `iperf3 -w 1M` bypasses it.
- With `-w 1M`: **356 Mbit/s (~44 MB/s), Cwnd 2.0 MB, ~6 retr in 424 MB** —
  window scaling live end-to-end; path capacity ≈ the OVH plan's public
  bandwidth. 14× the clamped throughput.
- Firebird/Node sockets set no explicit buffers → genuine autotuning applies;
  awaiting the user's re-run of the custom bench (fetch conns = 1, previously
  2.8 MB/s) for the end-to-end confirmation. Driver pipeline (32-deep,
  ~2 MiB in flight) comfortably covers ~100 MB/s at this RTT.

### End-to-end confirmation (user re-ran custom bench, remote, 100×1.7 MB)
- Fetch, 1 conn: **2.8 → 10 MB/s** (55.6 s → 16.7 s for ~167 MiB). One
  connection now outruns yesterday's 8-connection aggregate (10.4).
- Path offers ~44 MB/s (iperf −w 1M); the gap is now per-blob round-trip
  overhead (open + close + drain ≈ 2–3 RTTs per blob around a ~40 ms
  transfer → ≈10 MB/s at this RTT). Levers: fetch conns 4/8 today;
  cross-blob pipelining (overlap open/read/close across rows on one
  connection) as future driver work → added to the deferred backlog.

### Final numbers (fetch conns = 8, after everything)
- **166.9 MiB in 2.81 s = 59.4 MB/s** (vs ~100 s / ~2 MB/s at the start of the
  investigation — **~35× end-to-end**). 8 parallel streams exceed the 44 MB/s
  single-stream iperf figure, as expected. Stack-up: blob segment pipelining
  (driver) × Tcp1323Opts fix (server registry) × pool.map parallel fetch.
