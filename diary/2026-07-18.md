# 2026-07-18 — Session 41: FB6/protocol 20 + node-firebird 2.x reality check

Trigger: Marcio spotted node-firebird v2.10.0 release notes (FB6 protocol
support). Verified against the npm registry: the whole 2.x modernization is
real and FAST — 2.4.0→2.10.0 shipped Jul 13–17 (TypeScript 7 migration,
promises, AbortSignal, batch API, pool with metrics, statement LRU cache,
queryStream, Srp256/384/512, ChaCha, protocol 20). Three consequences: our
README positioning ("callback era / very old Node retro-compat") had gone
stale within a day of writing it; the benchmark pin (^1.1.9) would have been
indefensible at launch; and FB6/P20 became a feature gap.

## Protocol 20 (FB6)

Researched from the FB6-era source in `references/firebird` (protocol.h,
protocol.cpp, server.cpp):

- v20's only client-relevant wire change: `op_prepare_statement` /
  `op_exec_immediate` gain a TRAILING `p_sqlst_flags` xdr short (4 bytes on
  the wire) — `IStatement::PREPARE_*` values; 0 is valid (server ORs them
  with info-item-derived flags). A v20 server BLOCKS reading the prepare
  without the field — this was node-firebird's long-standing "prepare hang".
- Pre-FB6 servers read at most 10 `p_cnct_versions` offers (we now send 6).

Implementation: `PROTOCOL_VERSION20` constant + SUPPORTED_PROTOCOLS entry
(weight 6); one gated `int32(0)` in `allocateAndPrepare`. That's the whole
driver change — the ≥19 inline-blob gates carry over naturally since 20 ≥ 19.

Docker: `fb6` lane (`firebirdsql/firebird:6-snapshot`, port 30507, aux 3057)
in the compose matrix; both test envs got `{ name: 'fb6', port: 30507,
version: 6 }` with an `FB_SKIP_FB6=1` opt-out (snapshot images move). CI
inherits the lane via the shared compose file.

Test result: **1016 core + 88 drizzle green on FB3/4/5/6**. Fallout was two
test-expectation fixes in `inline-blobs.test.ts` (a `version === 5` gate that
should be `>= 5` — renamed `hasInlineBlobs` — and the protocol-negotiation
map gaining `6: 20`). FB6 passes the full inline-blob behavior suite:
zero-RT small blobs, thresholds, tx-scoped cache. Smoke: engine 6.0.0,
protocol 20, blob round-trip, 22 RTs for the whole session.

## Benchmarks vs node-firebird 2.10.0

Bumped `^1.1.9` → `^2.10.0`. Their callback API (attach/query/transaction/
blob-as-callback) survived 2.x unchanged, so the adapter needed only the
removal of the `pluginName: 'Srp'` workaround (2.x has Srp256; both drivers
now run at out-of-the-box defaults). Probe gotcha: a transient blob (`cast(…)
from rdb$database`) read after the implicit tx commits raises "Invalid BLOB
ID" — table blobs (what the bench uses) are fine.

Results (defaults vs defaults, FB5): the story HOLDS against 2.x — their
sprint didn't touch the wire mechanics that dominate. Blobs 21×/115×/152×
(1MB) and 33×/124×/140× (300×8KB) at 0/2/10ms — slightly wider than vs
1.1.10 at 0ms; their blob path is still sequential per-blob open/read/close
in 1KB chunks. Open-tx selects 3.0–3.1×, inserts 1.6–1.9×, scans 1.1–1.7×.
One loss: bare connect at 0ms (0.8×, wireCrypt default). Full table in
plans/performance.md "Results 2026-07-18".

## Addendum — FB6 feature survey + JSON canary

Live-probed the 6.0.0 snapshot: schemas work (CURRENT_SCHEMA=PUBLIC,
RDB$SCHEMAS, system tables in SYSTEM schema), UNLIST / GREATEST / LEAST /
BTRIM / ANY_VALUE work; json_object / json_array / json_arrayagg all raise
syntax errors and parse.y has zero JSON tokens — node-firebird 2.4.0's
"native JSON" release note is overstated. New DPB items confirmed in source:
isc_dpb_owner=102, isc_dpb_search_path=105 (the tags node-firebird 2.7–2.9
mis-mapped).

Backlog #14 shipped same session: `searchPath` (string|string[] → DPB 105,
sent on attach+create when set) and `owner` (DPB 102, create only) connect
options. Empirical surprise: pre-FB6 servers IGNORE unknown DPB tags (FB3 and
FB5 attach fine with search_path set) — the initial "rejects" assumption was
wrong and got corrected in code comments/docs before it shipped. FB6
verification: `RDB$GET_CONTEXT('SYSTEM','SEARCH_PATH')` echoes the path with
SYSTEM auto-appended, CURRENT_SCHEMA follows the first path entry,
`mon$database.mon$owner` shows the owner. 7 new integration tests
(fb6-attach-options.test.ts) + README "Firebird 6" section.

Added `fb6-json-canary.test.ts` (drizzle suite, fb6 lane only, skipIf’d):
asserts JSON functions are ABSENT — the day a snapshot ships json_arrayagg
CI goes red with a message pointing at plans/drizzle.md, which is the
trigger to build the capability-gated single-query relational mode. 90
drizzle tests green (88 + 2 canary). Roadmap backlog #14–16 added:
searchPath/owner attach options (small, near-term), JSON-gated relational
mode (canary-triggered), TS 6/7 upgrade (parked until after v0.1.0 — the
d.ts pipeline is the risk, not the typecheck).
