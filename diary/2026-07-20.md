# 2026-07-20 — Session 43: named parameters (`@name`)

## Sigil choice: `@` over `:`

node-firebird 2.9.0 shipped `:name` named placeholders (mysql2's
`namedPlaceholders`). We deliberately diverged to `@name`. The reason is
Firebird-specific: a leading colon is already meaningful in PSQL — `:var` is
a **local-variable reference** (`SELECT … INTO :v`, `WHERE x = :v` inside an
EXECUTE BLOCK / stored-procedure body). A `:name` rewriter has to special-
case PSQL bodies or risk corrupting valid statements, and even then a typo'd
name silently degrades into a variable reference. `@` has zero meaning
anywhere in Firebird's SQL or PSQL grammar, so `@name` is unambiguous even
sitting next to a real `:var` inside an EXECUTE BLOCK. One rule, no
exceptions, no PSQL carve-out — the docs can just say "works everywhere."

## Implementation

`packages/core/src/api/named-params.ts`: a scanner that rewrites `@name` →
`?` and returns the names in `?` order. It reuses the script parser's lexing
discipline — string literals, `"quoted idents"`, `q'{…}'` literals, and
`--` / `/* */` comments are skipped so a `@` inside any of them is left
verbatim. Design points:

- **Object = named, array = positional.** `isNamedParams` is just
  `!Array.isArray`, so existing positional call sites are untouched by
  construction. A repeated `@x` emits one `?` per occurrence (Firebird has
  no parameter reuse) and binds the value into each slot.
- **Single funnel.** Rewrite+bind happens in `normalizeParams`, called at the
  two internal execution entry points (`runStatement`, `streamRows`), so
  query/queryOne/run/execute/queryStream and the pool wrappers inherit it
  for free — only their TS signatures widened to `QueryParams`.
- **Prepared statements** rewrite once at prepare time (the `?` SQL is what
  the server prepares/caches) and carry the ordered names on the
  `PreparedStatement`, so `stmt.query({…})` binds per-run without re-lexing.
- **Fail loud, not silent.** Missing name → `FirebirdParamError` listing
  every absent `@name`; a named object against `?`-only SQL also throws
  (catches the "wrong param style" mistake). Extra keys are ignored; an
  explicit `null`/`undefined` value is honored (bound as NULL) via an `in`
  check rather than truthiness.

## Usage

```ts
// Object = named params; key order is irrelevant (driver reorders to the SQL).
const rows = await db.query(
  'select * from emp where dept = @dept and sal > @min',
  { dept: 10, min: 5000 },
);

// execute / update / stream / pool — all inherit it from the shared funnel.
await db.execute('update emp set name = @name where id = @id', { name: 'Ann', id: 1 });
for await (const r of db.queryStream('select * from emp where dept = @dept', { dept: 10 })) { /* … */ }
await pool.query('select * from emp where id = @id', { id: 7 });

// Prepared: rewritten once, re-run with different objects.
const stmt = await db.prepare('select * from emp where dept = @dept and sal >= @floor');
await stmt.query({ dept: 10, floor: 2000 });
await stmt.query({ dept: 20, floor: 3000 });
await stmt.close();

// A name repeated in the SQL is bound into every slot (Firebird has no
// param reuse — the value goes on the wire once per occurrence):
await db.query('select * from t where a = @x or b = @x', { x: 10 }); // → [10, 10]

// Positional arrays keep working unchanged — pick per call.
await db.query('select * from emp where id = ?', [7]);

// Mixing throws FirebirdParamError:
//   - named object against ?-only SQL
//   - a @name the object doesn't provide
// null/undefined values are honored (bound as NULL); extra keys are ignored.
```

# 2026-07-20 — Session 44: wire batch API (`executeBatch`, backlog #17)

## What shipped

`executeBatch(sql, rows, opts?)` on Attachment / Transaction / Pool /
PreparedStatement: bulk DML over Firebird's wire batch API
(op_batch_create/msg/exec, FB4+/protocol ≥ 16). Design in `plans/batch.md`;
wire layer `src/protocol/batch.ts`, orchestration `src/api/batch.ts`.

```ts
const r = await db.executeBatch(
  'insert into orders (id, customer, total) values (?, ?, ?)',
  [[1, 'ACME', '99.90'], [2, 'Globex', '15.00']],
);
r.rowsAffected;   // 2
r.updateCounts;   // [1, 1]

// Rows can be @name objects and ANY (async) iterable — streamed, chunked
// every ~8 MiB automatically:
await db.executeBatch('insert into t (a, b) values (@a, @b)', rowsFromFile());

// Per-row errors instead of throwing at the first failure:
const r2 = await db.executeBatch(sql, rows, { continueOnError: true });
r2.errors; // [{ index, error }]

// Prepared: repeat batches skip the prepare round trip:
const stmt = await db.prepare(sql);
await stmt.executeBatch(rowsA); await stmt.executeBatch(rowsB, tx);
```

## The design problem: batch fixes ONE format

Our normal execute path sends a value-driven BLR per call (each `?` described
by the value it carries). A batch parses the BLR **once** at op_batch_create
and every row must match it. So the batch format is built from the
statement's **described inputs**, mapped to the wire types the server already
coerces from in our normal path: TEXT/VARYING → `blr_varying(desc.length)`
(no client-side CHAR padding), DATE/TIME/TIMESTAMP → `blr_timestamp` (server
truncates), EX time zones → non-EX, SHORT → `blr_long`. Scaled
NUMERIC/DECIMAL params are converted client-side with exact decimal-STRING
arithmetic (`decimalToScaled` — no float multiply, half-away-from-zero on
excess digits), so `'12.5'` at scale −2 is exactly 1250.

Each packed message on the wire is byte-identical to the protocol-13 execute
message (null bitmap + non-NULL values, XDR) — verified against
`xdr_packed_message` in firebird v5.0.1 protocol.cpp.

## Traps found (empirically, against real servers)

- **`p_batch_msglen` uses PADDED struct sizes for TZ types.** The engine
  validates our declared message length against its layout for our BLR
  (jaybird's calculateBatchMessageLength table). Sum-of-members says
  TIMESTAMP_TZ = 10, TIME_TZ = 6; the engine uses `sizeof(ISC_TIMESTAMP_TZ)`
  = **12** and `sizeof(ISC_TIME_TZ)` = **8** (trailing USHORT padded to the
  struct's 4-byte alignment). Wrong values → SQLDA error −804 at create.
  Found by a per-type bisect harness on FB4.
- **Blob params need `blr_blob2`, not `blr_quad`.** DsqlBatch scans the
  batch format for blob-typed fields; quad hides them, and
  op_batch_regblob then fails with "no blobs in associated with batch
  statement". Emitting blob2 (subtype + charset words) fixes it. Blobs are
  uploaded with the existing writeBlob machinery and registered as
  themselves (jaybird-style, BLOB_ID_USER; the remote server rewrites the
  policy to BLOB_STREAM internally).
- **Response accounting instead of deferred-drop.** create/msg/regblob each
  answer with an op_response that arrives before op_batch_cs (both sides are
  lazy-send, so create+msgs+exec still cost ONE flush). We read them
  explicitly in order — a create/msg failure surfaces as its real error —
  and on failure mark the still-inbound replies deferred so the connection
  stays in sync (same pattern as executeStatement's piggyback error path).

## Round trips (the point of the feature)

Cached statement, 300 rows: **3 flushes total** — tx start, one
create+msg+exec packet, commit (asserted via `db.roundTrips` in the test).
The per-statement alternative costs 300+. Chunking splits cycles every
~8 MiB (server's default batch buffer is 16 MiB, hard cap 256 MiB), so
memory stays bounded for streamed sources and `isc_batch_too_big` is
structurally unreachable.

## Errors & semantics

Default: first failed row throws `FirebirdBatchError` (`.index`, `.result`)
and the own-tx wrappers roll back — nothing commits. `continueOnError: true`
sets MULTIERROR: resolves with `errors: [{index, error}]` (error null beyond
the TAG_DETAILED_ERRORS cap, default 64) and commits the good rows.
`updateCounts` per processed row (−1 = failed). Client-side rejections
(RETURNING, no parameters, out-of-range ints, oversized Buffers) name the
row and parameter; server-side length checks (e.g. varchar CHARACTER counts
under a UTF8 connection — the client can only cap BYTES) surface as the
server's truncation error with the connection intact.
