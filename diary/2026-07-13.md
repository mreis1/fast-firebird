## Session 23 — blob/memo semantics Q&A, abandoned-stream fix, charset-NONE test gaps closed (work spans 07-11 → 07-13)

User asked how text blobs (memos) are handled today; auditing the answers
surfaced one real defect and two test gaps. All fixed.

### How memos work (verified in code, now fully asserted)
- `SELECT MEMO` is **eager by default**: subtype-1 blobs materialize during
  row decode and arrive as decoded **strings** (column charset codec); binary
  blobs arrive as Buffers.
- Lazy mode is type-blind: `blobs:'lazy'` (connection or per-query) turns
  every blob column into a `Blob` handle — memos included, nothing is
  permanently eager. Handles are tx-scoped; auto-tx `db.query` rejects lazy.
- CHARSET NONE memos decode via `charsetNoneEncoding` (blob descriptor keeps
  the charset id in `scale & 0xff`), both directions (string params encode
  through the same codec).

### Fix: abandoned blob streams leaked the server handle
`Blob.stream()` only issued `op_close_blob` on EOF. An early `destroy()`, a
`break` out of `for await`, or the error path left the handle open until tx
end. Now the Readable has a `destroy` handler that queues the deferred close
through the op-lock (serializes behind any in-flight pull); guards prevent a
mid-flight pull from pushing to the dead stream or double-closing.
Observability trick for the tests: `wire.deferredResponses` — the stream's
`openBlob` flush consumes all previously pending deferred responses, so
mid-stream the counter is deterministically 0 and the abandon-close brings it
to exactly 1. Three new tests (100 KB blob): magic-number pattern
(read head → break → close queued → wire still in sync for a full re-read),
destroy-before-first-read (no handle opened, no close sent), and
tx-closed-under-a-paused-stream (FirebirdBlobError, connection usable after).

### Charset-NONE test gaps closed
The €/smart-quotes round-trip only covered the *eager* memo path. Added:
lazy `Blob.text()` through `charsetNoneEncoding`, and `charsetOverrides`
keyed on the blob column itself (`HISTORY.LONG_MEMO`). Both green FB3/4/5 —
codec paths were already correct, just unasserted.

## Session 25 — expandStar shipped, lazy-binary/lazy-text modes, offline connection editor

### `expandStar` shipped — backlog item 1 (`SELECT A.*, TABLE.* FROM TABLE`)
`{expandStar: true}` rewrites top-level `*`/`alias.*`/`table.*` into an
explicit column list (minus `exclude`, filtered by `only`) BEFORE preparing —
excluded scalars are genuinely never sent (decode-exclude only ever saved
blob reads). How:
- Describe now also requests **isc_info_sql_relation_alias** → every output
  column knows its FROM alias. `a.*` maps to its exact output run — self-joins
  (`t1.*, t2.*` on the same table) expand correctly, no FROM parsing at all.
- Tokenizer (string/q-literal/quoted-ident/comment aware, paren depth) finds
  top-level stars only: `count(*)`, `a*b`, `'*'`, subquery/CTE stars untouched;
  FIRST/SKIP/DISTINCT prefixes handled.
- Emitted names always double-quoted → `"weird col"` survives.
- Rewrites cached per (sql, only, exclude); invalidated with the statement
  cache on DDL (asserted: `alter table add column` shows up on next run).
- RowMapper exclude/only now also match qualified `ALIAS.COL` entries so
  rewrite-time and decode-time filters compose.
- Clear errors: top-level UNION; excluding every column. Learned live: a BARE
  `*` mixed with other select items is invalid Firebird SQL (prepare rejects
  "token unknown *") — only qualified stars mix, so the rewriter never sees
  that case.
- Wire-truth test trick: `rowMode:'array'` ignores decode-time exclude — if
  the column still disappears, it was cut from the statement itself.

## Session 26 — cross-blob pipelining SHIPPED (deferred backlog #3, both halves)

Usage examples validated with the user first (eager automatic, no flag;
`blobReadAhead: true | depth | {columns, depth, maxBytes}` + connection-level
default; 16 MiB budget; buffered-lazy stays on-demand).

### Part A — eager overlap (`readBlobs`, automatic)
Batch materialization now reads ALL of a batch's blobs through one scheduler:
a single FIFO pending queue over interleaved open/segment/close requests, a
GLOBAL 32-deep segment window spanning as many blobs as needed, opens issued
10 ahead, writes batched (flush at ≥8 requests or empty pipe), per-blob
window cap ramped 4→32 only when a full window returns pure data (small blobs
≈ zero over-reads, big blobs reach full depth). Error drain discipline: shift
descriptor before await; on error flush held bytes, drain every response,
deferred-close open handles.

Three bugs found by the tests en route — worth remembering:
1. **closeBlobDeferred is unsafe with responses in flight** — the deferred
   mechanism discards the FIRST response that arrives, not the close's own;
   fine everywhere else (nothing outstanding), fatal inside the pipeline
   ("invalid BLOB handle" chaos). Closes are now in-order pending descriptors.
2. Flushing after every response = 1 packet per top-up (~180 flushes for 80
   blobs). Batched flushes cut it to <1/blob — but held-back request BYTES
   already sit in the writer, so the error path must flush + drain them too.
3. head/head+1 window sharing stalls small blobs (2 blobs per RTT); the
   global window streams ~8 small blobs per RTT.

Measured (in-process latency proxy, 30 ms RTT, 20×200 KB): 967→425 ms,
48→21 ms/blob — under 1 RTT per blob; serial was ~135 ms/blob.

### Part B — `blobReadAhead` for lazy queryStream
`BlobPrefetcher`: background fetch of upcoming rows' lazy blobs (row-order
queue, watermark = current row + depth, `maxBytes` budget with one-blob
overshoot, claimed fetches bypass budget/close) in bounded op-lock slices so
consumer ops interleave. `Blob.buffer()/stream()` serve from the prefetch
store via `scope.prefetch.take()` (single-shot handover frees budget).
Best-effort by design: skips/out-of-order/budget → silent on-demand fallback;
errors surface only on the actual read.

Bug found live: the first prefetcher used serial `getBlobSegment` (1 RTT per
64 KB!) — readAhead made the exporter SLOWER (2.7→3.7 s). New
`readBlobWindow` (8 segments per flush, 1 RTT per 512 KB slice) fixed it:
exporter loop at 30 ms RTT + 20 ms simulated fs write: **136→82 ms/row**.

Test-measurement lesson: "0 round trips" assertions must use a 2-row result
set — otherwise the background prefetch of row N+2 increments the shared
flush counter and looks like a false failure.

27 read-ahead + 12 pipeline tests; suite **539 green ×3 stable** on FB3/4/5.

## Session 27 — `blob.head(n)` + forward-resume cursor (deferred backlog #5)

The magic-number primitive: `await blob.head(8)` reads the first bytes (raw,
no charset decode) in 2 flushes (open + one pipelined window via
`readBlobWindow`) and KEEPS the server handle open at its position. The Blob
now carries a forward cursor (`consumed` chunks + open handle + eof flag):

- `buffer()` resumes from the cursor — RT-asserted: ≤1 flush after a head()
  (a fresh read costs 2+, so ≤1 proves no re-open); prefix + tail concat is
  byte-exact, then cached as usual.
- Widening `head(m > n)` reads only the delta; a head that hits EOF promotes
  the cursor to the cache (subsequent buffer() = 0 flushes).
- `stream()` takes OVER the cursor (one-shot): pushes the consumed preamble
  first (with backpressure), then continues pulling from the transferred
  handle; the existing abandon-destroy close applies to it. Asserted: full
  content exactly once, strictly fewer flushes than a fresh stream.
- `blob.close()` releases an unfinished cursor (deferred close) and resets —
  a later read starts clean from byte 0. Unfinished cursors otherwise die
  with the transaction (asserted: interleaved queries fine, tx close fine).
- Prefetched blobs (blobReadAhead) serve `head()` from the store: 0 flushes.
- `text()` after a memo head still resumes AND charset-decodes correctly.

9 scenarios × FB3/4/5 = 27 tests, all first-try green; suite **566 green**.

## Session 29 — blob write streaming (deferred backlog #6): the last read/write asymmetry

`ParamValue` now accepts `AsyncIterable<Buffer | string>` (covers
`fs.createReadStream`, `Readable.from`, async generators) for BLOB columns:

```ts
await db.execute('insert into files (id, doc) values (?, ?)',
  [1, fs.createReadStream(path)]);   // never buffers the file in the driver
```

- `writeBlobStream`: create_blob2 → accumulate arbitrary chunk sizes into
  wire-max segments (merge small, slice big) → pipelined op_batch_segments
  window (same PIPELINE_DEPTH discipline as writeBlob) → close. String
  chunks encode through the column's charset codec (memo streaming with €
  asserted).
- **Bug caught by the error-path test (hang, not failure!)**: a source that
  throws BETWEEN writes leaves segments written-but-unflushed; the drain then

  handle directly → rejected with guidance ("pass `await blob.buffer()`"):
  its stream()/buffer() would re-enter the op-lock we already hold —

- Multiple stream params in one statement, empty sources (empty non-NULL

## Session 31 — FB5 inline blobs (protocol 19): small blobs are now FREE

The last blob-round item, and the deepest: first protocol-version bump since
M1. Studied the reference implementation in references/firebird first.

### Protocol audit (P16 → P19) — smaller than feared
- Version-gated fields for ops we send: op_execute gains timeout(≥16),
  cursor_flags(≥18), inline_blob_size(≥19) — and executeStatement had been
  future-proofed with exactly those conditionals in an earlier pass, so only
  the inline size value needed wiring. op_prepare changes only at ≥20 (we
  stop at 19). op_fetch unchanged. Crypt/auth version-agnostic (verified: no
  `=== 16`-style comparisons anywhere).
- We deliberately DON'T offer 17 (op_batch_sync) / 18 (op_fetch_scroll):
  nothing we send uses them. Negotiations: FB3→15 (not 13! it supports
  crypt-callback-at-connect), FB4→16, FB 5.0.2+→19 (docker fb5 = 5.0.4).

### Wire format (from fbserver source, sendInlineBlob/setupInlineBlob)
op_inline_blob(114): tran id (int32) + blob id (8 bytes) + blob info
(counted opaque, clumplets) + blob data (counted opaque) — and the data
payload is **UInt16LE-segment-framed, identical to op_get_segment bodies**,
so the existing frame parser applies verbatim. Packets arrive BEFORE the
op_fetch_response/op_sql_response they accompany; server sends them only for
blobs whose (info + framed data) fits the size the CLIENT announces in each
op_execute (P19 field; fbclient default 65535, 0 = off). Skipped blobs fall
back to the normal pipeline transparently.

### Our implementation
- `readOp()` (the single funnel every response path goes through) consumes
  op_inline_blob packets and hands them to `wire.onInlineBlob` — transparent
  to fetch loops, execute paths, blob pipelines, everything.
- `InlineBlobCache` (tx-scoped, fbclient-mirroring semantics): single-shot
  take() (re-reads fall back to the wire), byte budget (`maxBlobCacheSize`,
  10 MiB default) with silent drop on overflow, cleared on commit/rollback/
  disconnect.
- Every blob read path consults it FIRST: eager materialize, lazy
  buffer()/text()/head()/stream(), BlobPrefetcher, prefetchBlobs.
- Options: `maxInlineBlobSize` (default 65535, clamp 0–65535, 0 = off),
  `maxBlobCacheSize`.

### Tests: the feature broke three tests BY WORKING
Full-suite run with P19 live: exactly 3 FB5-only failures — suites whose
premise was "reading small blobs costs round trips" (lazy-vs-eager RT,
read-ahead control, prefetch-vs-serial comparison). Fixed by disabling
inline (`maxInlineBlobSize: 0` via new freshDb extra param) in suites that
measure the explicit wire machinery. New inline-blobs suite (12 scenarios ×
FB3/4/5 = 36): protocol negotiation per server, eager fetch of 20 rows ×
(4 KB doc + memo) costs ≤ baseline+1 flushes on FB5 (vs strictly more on
FB3/4), mixed small+big row, all four lazy read methods at 0 RT, prefetchBlobs

wire), 0-disables, tiny cache budget stays correct, tx-scoped cleanup
(cache empty after commit), read-ahead composition, NULLs + expandStar.

Suite: **674 green ×2 stable** on FB3/4/5; drizzle tsc clean; dist rebuilt.

  snapshot (name/version/deps) so the frozen install resolves without it.
- **Build order**: drizzle's typecheck resolves core via its BUILT type
  declarations — core must build before drizzle typechecks. Invisible on a
  dev machine where dist/ always exists; a guaranteed red X on GitHub.
