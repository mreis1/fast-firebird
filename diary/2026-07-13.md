# 2026-07-13

## Session 23 — blob/memo semantics Q&A, abandoned-stream fix, charset-NONE test gaps closed (work spans 07-11 → 07-13)

User asked how text blobs (memos) are handled today; auditing the answers
surfaced one real defect and two test gaps. All fixed.

### How memos work (verified in code, now fully asserted)
- `SELECT MEMO` is **eager by default**: subtype-1 blobs materialize during
  row decode and arrive as decoded **strings** (column charset codec); binary
  blobs arrive as Buffers.
- Lazy mode is type-blind: `blobs:'lazy'` (connection or per-query) turns
  every blob column into a `Blob` handle — memos included, nothing is
  permanently eager. Handles are tx-scoped; auto-tx `db.query` rejects lazy.
- CHARSET NONE memos decode via `charsetNoneEncoding` (blob descriptor keeps
  the charset id in `scale & 0xff`), both directions (string params encode
  through the same codec).

### Fix: abandoned blob streams leaked the server handle
`Blob.stream()` only issued `op_close_blob` on EOF. An early `destroy()`, a
`break` out of `for await`, or the error path left the handle open until tx
end. Now the Readable has a `destroy` handler that queues the deferred close
through the op-lock (serializes behind any in-flight pull); guards prevent a
mid-flight pull from pushing to the dead stream or double-closing.
Observability trick for the tests: `wire.deferredResponses` — the stream's
`openBlob` flush consumes all previously pending deferred responses, so
mid-stream the counter is deterministically 0 and the abandon-close brings it
to exactly 1. Three new tests (100 KB blob): magic-number pattern
(read head → break → close queued → wire still in sync for a full re-read),
destroy-before-first-read (no handle opened, no close sent), and
tx-closed-under-a-paused-stream (FirebirdBlobError, connection usable after).

### Charset-NONE test gaps closed
The €/smart-quotes round-trip only covered the *eager* memo path. Added:
lazy `Blob.text()` through `charsetNoneEncoding`, and `charsetOverrides`
keyed on the blob column itself (`HISTORY.LONG_MEMO`). Both green FB3/4/5 —
codec paths were already correct, just unasserted.

## Session 24 — eager-by-default locked, ColumnInfo metadata, demo charset/role pickers

### Design decision (final, user verdict): blobs stay EAGER by default
Recorded in the roadmap's new "Design decisions (locked)" section, motivation
verbatim: "SELECT gives me values" is the near-universal driver expectation,
and lazy leaks transaction-lifetime concerns into ordinary code. Also:
flipping would break every consumer, and forced-eager `db.query` would make
row shape depend on the call site. Lazy remains one option away
(`connect({blobs:'lazy'})` or per-query).

### Core: `QueryResult.columns` (ColumnInfo[])
Answers "what did `select *` actually return?" — previously a mystery in
`rowMode:'array'`. Per kept column, in row/position order: `name` (the row
key / positional header, alias-aware, case per `lowercaseKeys`), `field`
(source field behind an alias), `relation` (undefined for expressions),
`type` (friendly names: VARCHAR, INTEGER, NUMERIC/DECIMAL for scaled ints,
BLOB SUB_TYPE TEXT, …), `nullable`. Object mode lists kept columns after
`only`/`exclude`; array mode lists all positionally. Built in RowMapper (it
already had per-column state); exported `ColumnInfo`; 2 new tests incl. the
`select *` + array-mode case. `PreparedStatement.outputs` still exposes raw
descriptors for streaming callers.

### Demo: charset picker + transcoder presets + role
- `ServerConfig` gains `charset` (default NONE), `charsetNoneEncoding`
  (default win1252 — behavior unchanged for existing setups) and `role`.
- `connectOptsFor` remains the single builder: transcoder only applies when
  charset is NONE; role passes through to core and the compat manager.
- `/api/servers/:id/config` patches charset/transcoder/role (connect-time DPB
  settings → tears down state; panels remount via the existing nonce).
- Web: reusable `CharsetPicker` (charset select; for NONE a "decode as"
  preset select — win1252/latin1/cp1250/cp1251/utf8 — plus custom… free-text
  iconv name). Wired into AddServer (with a role input) and InfoPanel, which
  now shows charset ⇢ transcoder + role rows with an inline
  "change → apply & reconnect" editor.
- Verified end-to-end against the live dev server: NONE→UTF8→NONE round-trip,
  and role actually reaching the server (`CURRENT_ROLE` = a granted
  DEMO_ROLE; note an *ungranted* role is silently ignored by Firebird —
  CURRENT_ROLE stays NONE — worth remembering when a user reports "role
  doesn't work").

### Answered along the way
- Fetch sizing: adaptive — first batch 40 rows (`RAMP_START_ROWS`), doubling
  per batch toward min(`fetchSize` cap, 256 KB byte budget), floor 10. Blob
  cells cost 8 bytes in the width estimate (id only), so blob rows batch wide.
- Eager batches materialize all their blobs before yielding (memory = batch ×
  blob size); lazy rows carry only ids — the reason lazy exists.
- Lazy handles from any batch (incl. the last) are valid while the
  queryStream iteration is running; collecting them past the loop throws.
- Cross-blob pipelining backlog item extended with the lazy `blobReadAhead`
  half + FB5 inline-blob companion note; per-column blob mode and
  `blob.head()`/resume-cursor added as backlog items 4–5, small DX helpers
  (per-query fetchSize, queryOne, typed rows, asyncDispose) as item 8.
