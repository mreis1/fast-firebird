## Session 23 ‚Äî blob/memo semantics Q&A, abandoned-stream fix, charset-NONE test gaps closed (work spans 07-11 ‚Üí 07-13)

User asked how text blobs (memos) are handled today; auditing the answers
surfaced one real defect and two test gaps. All fixed.

### How memos work (verified in code, now fully asserted)
- `SELECT MEMO` is **eager by default**: subtype-1 blobs materialize during
  row decode and arrive as decoded **strings** (column charset codec); binary
  blobs arrive as Buffers.
- Lazy mode is type-blind: `blobs:'lazy'` (connection or per-query) turns
  every blob column into a `Blob` handle ‚Äî memos included, nothing is
  permanently eager. Handles are tx-scoped; auto-tx `db.query` rejects lazy.
- CHARSET NONE memos decode via `charsetNoneEncoding` (blob descriptor keeps
  the charset id in `scale & 0xff`), both directions (string params encode
  through the same codec).

### Fix: abandoned blob streams leaked the server handle
`Blob.stream()` only issued `op_close_blob` on EOF. An early `destroy()`, a
`break` out of `for await`, or the error path left the handle open until tx
end. Now the Readable has a `destroy` handler that queues the deferred close
through the op-lock (serializes behind any in-flight pull); guards prevent a
mid-flight pull from pushing to the dead stream or double-closing.
Observability trick for the tests: `wire.deferredResponses` ‚Äî the stream's
`openBlob` flush consumes all previously pending deferred responses, so
mid-stream the counter is deterministically 0 and the abandon-close brings it
to exactly 1. Three new tests (100 KB blob): magic-number pattern
(read head ‚Üí break ‚Üí close queued ‚Üí wire still in sync for a full re-read),
destroy-before-first-read (no handle opened, no close sent), and
tx-closed-under-a-paused-stream (FirebirdBlobError, connection usable after).

### Charset-NONE test gaps closed
The ‚Ç¨/smart-quotes round-trip only covered the *eager* memo path. Added:
lazy `Blob.text()` through `charsetNoneEncoding`, and `charsetOverrides`
keyed on the blob column itself (`HISTORY.LONG_MEMO`). Both green FB3/4/5 ‚Äî
codec paths were already correct, just unasserted.

## Session 26 ‚Äî cross-blob pipelining SHIPPED (deferred backlog #3, both halves)

Usage examples validated with the user first (eager automatic, no flag;
`blobReadAhead: true | depth | {columns, depth, maxBytes}` + connection-level
default; 16 MiB budget; buffered-lazy stays on-demand).

### Part A ‚Äî eager overlap (`readBlobs`, automatic)
Batch materialization now reads ALL of a batch's blobs through one scheduler:
a single FIFO pending queue over interleaved open/segment/close requests, a
GLOBAL 32-deep segment window spanning as many blobs as needed, opens issued
10 ahead, writes batched (flush at ‚â•8 requests or empty pipe), per-blob
window cap ramped 4‚Üí32 only when a full window returns pure data (small blobs
‚âà zero over-reads, big blobs reach full depth). Error drain discipline: shift
descriptor before await; on error flush held bytes, drain every response,
deferred-close open handles.

Three bugs found by the tests en route ‚Äî worth remembering:
1. **closeBlobDeferred is unsafe with responses in flight** ‚Äî the deferred
   mechanism discards the FIRST response that arrives, not the close's own;
   fine everywhere else (nothing outstanding), fatal inside the pipeline
   ("invalid BLOB handle" chaos). Closes are now in-order pending descriptors.
2. Flushing after every response = 1 packet per top-up (~180 flushes for 80
   blobs). Batched flushes cut it to <1/blob ‚Äî but held-back request BYTES
   already sit in the writer, so the error path must flush + drain them too.
3. head/head+1 window sharing stalls small blobs (2 blobs per RTT); the
   global window streams ~8 small blobs per RTT.

Measured (in-process latency proxy, 30 ms RTT, 20√ó200 KB): 967‚Üí425 ms,
48‚Üí21 ms/blob ‚Äî under 1 RTT per blob; serial was ~135 ms/blob.

### Part B ‚Äî `blobReadAhead` for lazy queryStream
`BlobPrefetcher`: background fetch of upcoming rows' lazy blobs (row-order
queue, watermark = current row + depth, `maxBytes` budget with one-blob
overshoot, claimed fetches bypass budget/close) in bounded op-lock slices so
consumer ops interleave. `Blob.buffer()/stream()` serve from the prefetch
store via `scope.prefetch.take()` (single-shot handover frees budget).
Best-effort by design: skips/out-of-order/budget ‚Üí silent on-demand fallback;
errors surface only on the actual read.

Bug found live: the first prefetcher used serial `getBlobSegment` (1 RTT per
64 KB!) ‚Äî readAhead made the exporter SLOWER (2.7‚Üí3.7 s). New
`readBlobWindow` (8 segments per flush, 1 RTT per 512 KB slice) fixed it:
exporter loop at 30 ms RTT + 20 ms simulated fs write: **136‚Üí82 ms/row**.

Test-measurement lesson: "0 round trips" assertions must use a 2-row result
set ‚Äî otherwise the background prefetch of row N+2 increments the shared
flush counter and looks like a false failure.

27 read-ahead + 12 pipeline tests; suite **539 green √ó3 stable** on FB3/4/5.

## Session 27 ‚Äî `blob.head(n)` + forward-resume cursor (deferred backlog #5)

The magic-number primitive: `await blob.head(8)` reads the first bytes (raw,
no charset decode) in 2 flushes (open + one pipelined window via
`readBlobWindow`) and KEEPS the server handle open at its position. The Blob
now carries a forward cursor (`consumed` chunks + open handle + eof flag):

- `buffer()` resumes from the cursor ‚Äî RT-asserted: ‚â§1 flush after a head()
  (a fresh read costs 2+, so ‚â§1 proves no re-open); prefix + tail concat is
  byte-exact, then cached as usual.
- Widening `head(m > n)` reads only the delta; a head that hits EOF promotes
  the cursor to the cache (subsequent buffer() = 0 flushes).
- `stream()` takes OVER the cursor (one-shot): pushes the consumed preamble
  first (with backpressure), then continues pulling from the transferred
  handle; the existing abandon-destroy close applies to it. Asserted: full
  content exactly once, strictly fewer flushes than a fresh stream.
- `blob.close()` releases an unfinished cursor (deferred close) and resets ‚Äî
  a later read starts clean from byte 0. Unfinished cursors otherwise die
  with the transaction (asserted: interleaved queries fine, tx close fine).
- Prefetched blobs (blobReadAhead) serve `head()` from the store: 0 flushes.
- `text()` after a memo head still resumes AND charset-decodes correctly.

## Session 31 ‚Äî FB5 inline blobs (protocol 19): small blobs are now FREE

The last blob-round item, and the deepest: first protocol-version bump since
M1. Studied the reference implementation in references/firebird first.

### Protocol audit (P16 ‚Üí P19) ‚Äî smaller than feared
- Version-gated fields for ops we send: op_execute gains timeout(‚â•16),
  cursor_flags(‚â•18), inline_blob_size(‚â•19) ‚Äî and executeStatement had been
  future-proofed with exactly those conditionals in an earlier pass, so only
  the inline size value needed wiring. op_prepare changes only at ‚â•20 (we
  stop at 19). op_fetch unchanged. Crypt/auth version-agnostic (verified: no
  `=== 16`-style comparisons anywhere).
- We deliberately DON'T offer 17 (op_batch_sync) / 18 (op_fetch_scroll):
  nothing we send uses them. Negotiations: FB3‚Üí15 (not 13! it supports
  crypt-callback-at-connect), FB4‚Üí16, FB 5.0.2+‚Üí19 (docker fb5 = 5.0.4).

### Wire format (from fbserver source, sendInlineBlob/setupInlineBlob)
op_inline_blob(114): tran id (int32) + blob id (8 bytes) + blob info
(counted opaque, clumplets) + blob data (counted opaque) ‚Äî and the data
payload is **UInt16LE-segment-framed, identical to op_get_segment bodies**,
so the existing frame parser applies verbatim. Packets arrive BEFORE the
op_fetch_response/op_sql_response they accompany; server sends them only for
blobs whose (info + framed data) fits the size the CLIENT announces in each
op_execute (P19 field; fbclient default 65535, 0 = off). Skipped blobs fall
back to the normal pipeline transparently.

### Our implementation
- `readOp()` (the single funnel every response path goes through) consumes
  op_inline_blob packets and hands them to `wire.onInlineBlob` ‚Äî transparent
  to fetch loops, execute paths, blob pipelines, everything.
- `InlineBlobCache` (tx-scoped, fbclient-mirroring semantics): single-shot
  take() (re-reads fall back to the wire), byte budget (`maxBlobCacheSize`,
  10 MiB default) with silent drop on overflow, cleared on commit/rollback/
  disconnect.
- Every blob read path consults it FIRST: eager materialize, lazy
  buffer()/text()/head()/stream(), BlobPrefetcher, prefetchBlobs.
- Options: `maxInlineBlobSize` (default 65535, clamp 0‚Äì65535, 0 = off),
  `maxBlobCacheSize`.

### Tests: the feature broke three tests BY WORKING
Full-suite run with P19 live: exactly 3 FB5-only failures ‚Äî suites whose
premise was "reading small blobs costs round trips" (lazy-vs-eager RT,
read-ahead control, prefetch-vs-serial comparison). Fixed by disabling
inline (`maxInlineBlobSize: 0` via new freshDb extra param) in suites that
measure the explicit wire machinery. New inline-blobs suite (12 scenarios √ó
FB3/4/5 = 36): protocol negotiation per server, eager fetch of 20 rows √ó
(4 KB doc + memo) costs ‚â§ baseline+1 flushes on FB5 (vs strictly more on
FB3/4), mixed small+big row, all four lazy read methods at 0 RT, prefetchBlobs
0 RT, single-shot re-read fallback, threshold honored (4 KB > 1 KB limit ‚Üí
wire), 0-disables, tiny cache budget stays correct, tx-scoped cleanup
(cache empty after commit), read-ahead composition, NULLs + expandStar.

Suite: **674 green √ó2 stable** on FB3/4/5; drizzle tsc clean; dist rebuilt.

## Session 33 ‚Äî zone-preserving time zone types (deferred backlog #2)

Opt-in `timeZones: 'zoned'` (connection-level ‚Äî column readers are cached
per statement, so a per-query override would fight the statement cache):
TIMESTAMP/TIME WITH TIME ZONE columns return `ZonedDate { date, zone }` ‚Äî
`date` is the exact UTC instant (identical to what default 'instant' mode
returns as a plain Date), `zone` is the stored IANA name or ¬±HH:MM offset.
`valueOf()` = epoch millis (sortable), `toString()` =
`2026-07-13T14:30:00.000Z[Europe/Lisbon]`. Wall-clock rendering is the
platform's job (`zd.date.toLocaleString(‚Ä¶, { timeZone: zd.zone })` ‚Äî Node's
Intl carries full tzdata; the driver deliberately ships none).

Mechanics:
- Zone ids, from the fbserver source: offset zones are
  `displacement + 1439` (ids 0..2878 ‚Üî ‚àí23:59..+23:59); named zones descend
  from 65535 through a fixed-order list ‚Äî generated
  `src/types/timezones.ts` (637 names, tzdata 2026b) from
  references/firebird `src/common/TimeZones.h`. Unknown future ids decode as
  `tz#NNNNN` and still round-trip.
- Decode: the wire already carried the id (an XDR short in a 4-byte slot we
  used to discard); TZ values are stored in UTC so the instant math is
  unchanged. EX variants' extra offset stays ignored (id is authoritative).
- Encode: `ZonedDate` is now a `ParamValue` ‚Äî sent as `blr_timestamp_tz`
  with a NEW UTC-based timestamp encoder (`encodeTimestampUtc`; the existing
  `encodeTimestamp` is local-wall-time for plain TIMESTAMP ‚Äî using it would
  have silently shifted instants by the client's UTC offset). Server coerces
  to TIME WITH TIME ZONE targets. Unknown zone names throw driver-side.
- Found & fixed en route: two literal NUL bytes had crept into session.ts
  (expandStar cache key ‚Äî meant `\u0000` escapes) making grep treat the file
  as binary.

Tests (9 scenarios √ó FB4/5 = 18; FB3 has no TZ types): instant-mode
unchanged, zoned instant+name, ¬±HH:MM offsets both signs, TIME WITH TIME
ZONE, table round-trip of zone AND instant (plus the same row read as plain
Date on an instant connection), WHERE-equality binding, plain-Date params
still accepted, unknown-zone error, SET TIME ZONE + CURRENT_TIMESTAMP.
Suite: **692 green**; drizzle tsc clean; dist rebuilt.

## Session 34 ‚Äî autoUpgradeReadOnly + DX helpers (deferred backlog #7 + #8)

Two backlog items in one pass, both pure api-layer work.

### autoUpgradeReadOnly (backlog #7)

Opt-in RO‚ÜíRW transaction auto-upgrade with statement replay, now in core
itself (the nf2-ext compat layer keeps its own copy at its level ‚Äî core's is
for direct users). Off by default: a write inside `readOnly: true` still
throws unless opted in via `TransactionOptions.autoUpgradeReadOnly` or the
connection-level default (per-tx wins, so `false` can override a `true`
connection).

Mechanics: `tx.run()` (thus query/execute) catches the failure, checks it is
genuinely isc_read_only_trans (gds 335544361, verified against the fbserver
sources; message-regex fallback), that the tx was started `readOnly: true`
and is still open ‚Äî then reuses the existing `restart()` machinery with
`{...options, readOnly: false}` and replays the statement ONCE. After the
upgrade `options.readOnly` is false, so a second failure propagates instead
of looping. Committing the RO tx during restart is trivially safe (it holds
no writes). Honest caveats, documented on the option: the upgrade is a real
commit + new transaction (snapshot moves forward; lazy Blob handles from
before it die via the existing generation bump), and only `query/run/execute`
replay ‚Äî `queryStream` and prepared statements don't. `tx.autoUpgraded`
reports that an upgrade happened.

Tests (7 √ó FB3/4/5 = 21): default-off error + tx-still-readable, per-tx
upgrade with same-tx visibility + commit persistence, no-second-upgrade for
later writes, connection-level default, per-tx false beats connection true,
PK violations never mistaken for a trigger, rollback-after-upgrade discards
the replayed write.

### Small DX helpers (backlog #8)

- **Per-query `fetchSize`** ‚Äî `QueryOptions.fetchSize` caps rows per fetch
  round trip for one statement (clamped 1‚Äì65535); the adaptive byte-budget
  plan still applies underneath, this just lowers/raises its ceiling. Wired
  into both `executePrepared` and `streamRows`.
- **`queryOne<T>()`** ‚Äî first row or `undefined`, on Attachment, Transaction,
  Pool and PreparedStatement. Doc is explicit that the full result set is
  still fetched ‚Äî put `FIRST 1` in the SQL when many rows could match.
- **Typed rows** ‚Äî `query<T>/run<T>/queryStream<T>/queryOne<T>` generics and
  `QueryResult<T = Row>`. Compile-time shaping only (casts at the API
  boundary, no runtime validation); `T` left unconstrained on purpose ‚Äî
  `interface MyRow` doesn't satisfy `Record<string, unknown>` (no implicit
  index signature on interfaces), so a constraint would reject the most
  natural usage.
- **`await using`** ‚Äî `Symbol.asyncDispose` on Attachment (disconnect),
  Transaction (rollback unless already finished ‚Äî dispose-without-commit
  means rollback), Pool (close) and PreparedStatement (close).
  `tsconfig.base.json` lib gained `ESNext.Disposable`; Node 22 has the
  symbol natively and vitest/esbuild lower the syntax, so the tests exercise
  real `await using` blocks. (Consumers on old `lib` settings are covered by
  the near-universal `skipLibCheck`.)
- **`blob.toFile(path)`** ‚Äî streams a lazy blob straight to disk through
  `stream()` + `pipeline` (no full buffering), returns bytes written.

Tests (13 √ó 3 = 39): queryOne row/empty/tx/prepared, typed rows incl.
streaming, fetchSize round-trip deltas on run + stream paths (tiny fetchSize
costs visibly more RTs, same rows), `await using` rollback-on-exit /
no-op-after-commit / attachment-disconnect / pool-close / statement-close,
toFile 300KB sha-exact + byte count.

clean, dist rebuilt. Roadmap: backlog #7 + #8 struck with ship notes;
status block refreshed (752 tests, CI done, README flagged as the debt).
