## Findings & fixes (chronological)

1. **SRP worked first try on all three servers** thanks to the research notes:
   Firebird's non-RFC quirks (proof mixes `H(N)^H(g) mod N` via modPow, exponent
   `(a+u·x) mod N`, session key always SHA-1/20 bytes, hex-encoded public keys,
   A-hex chunked 254 bytes into CNCT_specific_data) are all load-bearing.
2. **Docker images default WireCrypt=Required** → `FIREBIRD_CONF_WireCrypt: Enabled`
   in compose so both crypt modes are testable.
3. **FB3 + wireCrypt=disabled**: server sends op_accept_data with EMPTY data and
   answers op_attach with op_cont_auth — auth completes DURING attach. Eagerly
   sending op_cont_auth (node-firebird style) works on FB4/5 but FB3 replies
   "unavailable database". Implemented attach-time continuation (readResponseWithAuth).
4. **Error messages**: generated packages/core/src/protocol/messages.json (2539
   entries incl. SQLSTATE) from firebird/src/include/firebird/impl/msg/*.h via
   scripts/generate-messages.mjs. Code = 0x14000000 | facility<<16 | number.
5. **op_batch_segments has a p_sgmt_length field** before the segment cstring
   (protocol.cpp:481-486). Omitting it desyncs the stream → first hang. This is
   what node-firebird's mysterious `int32(len+2)` was.
6. **Boolean params**: SMALLINT→BOOLEAN has no implicit conversion in Firebird
   ("conversion error from string '1'"); send booleans as text 'true'/'false'
   (blr_text). Output blr_bool decodes fine as int32.
7. **Recursive CTE depth cap (1024)** — "Too many concurrent executions of the
   same request" from my 5000-row test SQL was a test bug, not a driver bug.
8. **Protocol ≥16 op_execute needs trailing p_sqldata_timeout int32** (≥18
   cursor flags, ≥19 inline blob size) — FB4/5 suites failed until added.
9. **Research-note erratum**: the firebird-core notes claimed wire INT64 is
   low-word-first; reading xdr.cpp shows the swap NORMALIZES to big-endian on
   the wire. INT64/INT128 are straight BE. (plans/research note stands corrected here.)
10. **Fetch batch termination confirmed from server.cpp:4340-4360**: every
    op_fetch batch ends with a terminator packet (status 0 = more, 100 = EOF,
    messages = 0) — safe to loop on it.
11. **One connection-level op lock** (promise chain) serializes wire ops; a
    hanging op poisons the chain — acceptable for now, add per-op timeouts in M3.

## Design decisions
- Value-driven BLR for parameters (NF-proven), described-type BLR for outputs.
- Blob cells decode to BlobRef during fetch and materialize AFTER the fetch
  stream completes (no op interleaving); subtype 1 → string via charset layer.
- INT64/INT128 'auto' mode: number when within safe range, else bigint.
- DECFLOAT: raw bytes placeholder (documented TODO).

## Session 2 — M3: statement cache & round-trip optimizations

### Shipped
- `WireConnection.flushCount` → `Attachment.roundTrips` (RT budgets are now
  test-asserted, not estimated).
- Execute+first-fetch (selects) and execute+record-counts (DML) coalesced into
  ONE packet; piggyback error paths drain via the deferred-response mechanism.
- Statement cache: LRU by SQL text, default 64 (`statementCacheSize`, 0 = off),
  deferred DSQL_close after cursor EOF, deferred DSQL_drop on eviction, cache
  cleared on same-connection DDL, re-prepare retry on gds 335544343.
- Public `PreparedStatement` (att.prepare → query/run/execute/close).
- RETURNING support: op_execute2 now used for insert/update/delete with outputs.
- Measured budgets (asserted on FB3/4/5): warm select ≤fetchSize rows = 1 RT,
  warm DML+count = 1 RT, cold = 2 RT, one-shot db.query warm = 3 RT.

### Findings
1. **Cached statements hold metadata existence locks**: `RECREATE TABLE` from
   another attachment BLOCKS while any attachment holds a prepared statement
   referencing the table. Normal Firebird ecosystem behavior (Delphi apps do
   this constantly) — documented in README with the `statementCacheSize: 0` escape.
2. **The server hard-closes the connection on an unknown statement handle**
   (verified FB3/4/5, both bogus and properly-dropped handles). So stale-handle
   errors can't be observed client-side; the retry path only fires for format
   errors. White-box stale test removed — it tested server behavior, not ours.
3. **SRP flake root-caused and fixed**: srp.cpp `computeScramble` uses
   `processStrippedInt` — u = SHA1(minimal(A) ‖ minimal(B)), NOT 128-byte
   padded as node-firebird notes suggested. Padded hashing failed whenever A or
   B was < 128 bytes (~1/128 of handshakes) with "user name and password are
   not defined". Also n2 = H(user) goes through BigInteger (stripped). k really
   is SHA1(N ‖ pad(g)) — the RemoteGroup ctor pads only there. Added
   deterministic regression (seed searched so A has a leading zero byte;
   internal `srpSeed` option) + 150-handshake soak. 8× full-suite runs clean.
4. vitest ran integration files in parallel → concurrent DDL deadlocks on FB3
   system tables; `fileParallelism: false` in vitest config.
5. `disconnect()` now swallows transport errors (dead connection) and always
   destroys the socket.

## Session 3 — WireCompression + benchmark harness

### Shipped

- **Benchmark harness** (`packages/benchmarks`): fast-firebird vs node-firebird
  1.1.10, in-process TCP `LatencyProxy` (one-way delay; no extra containers),

  300-insert / 1MB-blob at 0/2/10ms.

### Measured (FB5, defaults vs defaults; full table in performance.md)
- warm select (open tx): **~3× faster** at every latency (1 RT vs multi-RT).
- 10k scan **1.8×**, 300-insert **2×**, warm auto-tx select **1.7×** at 10ms.
- blob 1MB write+read: **14× (0ms) → 44× (10ms)** — node-firebird uses 1KB
  segments (~1000 RTs/MB); we use 64KB. Raised our default blob chunk to 64KB.
- connect+detach is ~0.7× (slower): we encrypt by default (op_crypt +1 RT +

### Findings
1. **Docker images ship WireCrypt=Required + only Srp256**: had to add
   `FIREBIRD_CONF_AuthServer: Srp256, Srp` and `WireCompression: true` to the

   attach-time continuation always works. Regression tests assert <5s failure.
3. `_processChunk` on a persistent zlib stream throws (`writeSync` on null
   handle) in Node 22 — must use the async event API.
4. node-firebird's own SRP flake (the padded-scramble bug) surfaced as ~4–13
   connect retries per benchmark run; harness retries and reports the count.

## Session 4 — ChaCha, adaptive fetch, pool, streaming

### Shipped
- **ChaCha/ChaCha64 wire crypt (FB4+)**: key = SHA256(20-byte SRP session key);
  IV is NOT in the op_crypt response (server sends it with `send_response(0,0)`)
  — it's pre-shared in the auth-completing op_response's p_resp_data as an
  UnTagged clumplet TAG_PLUGIN_SPECIFIC = `<plugin>\0<iv>` (remote.cpp
  addServerKeys). Parse that, engage crypt BEFORE reading the op_crypt response
  (which arrives encrypted). Node `chacha20` cipher, 16-byte IV = [ctr LE(4) |
  nonce(12)], verified against the RFC 8439 vector. `wireCryptPlugin` option;
  Arc4 stays default (universal); FB3 gets a clear "no ChaCha IV" error.
- **Adaptive fetch sizing** (`fetch-plan.ts`): rows/batch = clamp(256KB /
  estimated-row-width, 10, fetchSize), ramped 40→2×→cap across batches. 6000-row
  scan < 30 RTs; wide rows capped by budget; tiny queries don't over-fetch.
- **Connection pool** (`pool/pool.ts`): createPool/Pool, acquire/release/use,
  min/max, acquireTimeout, idle sweep to min, op_ping validate-on-borrow,
  per-connection statement cache. `Attachment.ping()`/`isAlive`,
  `Transport.isOpen`.
- **Row streaming** (`queryStream` on Attachment + Transaction): async generator
  over lazily-fetched batches (each batch fetched + blobs materialized under the
  connection op-lock, then yielded row-by-row), `finally` closes cursor +
  caches/drops stmt + commits/rolls-back the own-tx. Works with `Readable.from`.

### Findings
1. **ChaCha IV delivery**: modern FB4/5 (protocol 16) pre-shares the IV in the
   accept/auth op_response key clumplets, NOT in the op_crypt reply as
   node-firebird's code implies. The `keys` advertisement in op_cond_accept is
   empty on these builds; the IV rides the final auth op_response's p_resp_data.
2. **Pool capacity race (found by a test)**: a connection popped from idle for
   async `ping()` validation wasn't counted in `inUse` during the await, so
   concurrent acquires overshot `max` (peak 4 with max 3). Fix: reserve the slot
   in `inUse` before validating. The test asserted peak-concurrency, which
   caught it — a distinct-object count would have missed it.
3. **Server sends no IV in the op_crypt response** — `start_crypt` calls
   `send_response(sendL, 0, 0, ...)`; confirmed the IV must come from earlier.
4. **Test isolation**: streaming's 2500-insert beforeAll widened the window for
   Firebird's concurrent-DDL deadlock on system tables across the shared test
   DBs. Added `withRetry` (transient deadlock/conflict backoff) around setup DDL;
   forced single-fork sequential runner. 9× full-suite (200 tests) clean.
5. Wrong-password over plain Srp sometimes yields "connection closed" rather
   than an auth error (server drops the socket) — both are valid rejections.

### Result
200 tests green (unit + integration on FB3/4/5), 9 consecutive clean runs.
Both packages typecheck; core builds dual ESM/CJS.

## Session 7 — M5.5: lazy blobs, streaming, column exclude

### Shipped
- **Lazy blobs**: `blobs:'lazy'` (per-query or connection default; eager stays
  default) returns a `Blob` handle per non-null blob cell instead of
  materializing. `Blob.buffer()` (cached), `.text(enc?)` (subtype-1 via charset
  layer), `.stream({chunkSize})` (backpressured Readable, one op_get_segment per
  pull), `.size()` (op_info_blob). Verified on the 500-row-style scenario: read
  only the memo → unread blobs cost ZERO round trips (lazy ~1/3 the RTs of eager).
- **Dead-handle detection**: handles are transaction-scoped; reading one after
  its tx commits throws `FirebirdBlobError` (not a cryptic wire error). `db.query`
  with lazy throws (auto-tx would invalidate handles) — use tx.query/queryStream.
- **Column `exclude`/`only`**: decode-time projection — drops columns from the row
  AND skips materialization of excluded blobs (real RT savings in eager mode).
- **Connect timeout now covers the handshake+attach**, not just the TCP connect
  (a loaded server that accepts then stalls on responses no longer hangs forever).
- Protocol: split blob read into open/getSegment/close + blobTotalLength; the
  get_segment buffer is sized chunkSize+2 so tiny chunk sizes still progress
  (the 2-byte segment length prefix needs room).

### Findings
1. **Blob-id lifetime is the whole design**: ids are transaction-scoped, so
   lazy handles are only valid until commit. This drove the API — lazy requires
   an explicit tx or a stream; auto-tx `db.query` refuses it.
2. **Test flakiness root cause (finally nailed)**: beforeAll `recreate table`
   on the SHARED database ran in a wait-forever transaction; when another file's
   connection briefly held a metadata lock, the DDL blocked until the 60s hook
   timeout and cascaded to skips. Fix: a `ddl()` test helper runs DDL in a
   NOWAIT transaction and retries on lock conflict — transient contention now
   resolves in ms. 9× full-suite (290 tests) clean after the change.
3. **Tiny stream chunkSize hung**: a 2-byte read buffer had no room for the
   2-byte segment length prefix → 0 data forever. Fixed by requesting
   chunkSize + framing.

## Session 9 — M6: Drizzle adapter + a core param-coercion/wire-desync fix

Built `@fast-firebird/drizzle`. Firebird is Postgres-shaped, so the strategy is
"subclass pg-core, override the three things Firebird does differently":

- **FirebirdDialect extends PgDialect**
  - `escapeParam` → `?` (Firebird binds positionally on the wire).
  - `escapeName` → upper-fold then quote (`"USERS"`), matching Firebird's
    unquoted-name folding, so `pgTable('users', …)` hits table `USERS`.
  - `buildSelectQuery` → emit `SELECT FIRST n SKIP m …` instead of LIMIT/OFFSET
    (adapted from pg-core; private helpers reached via a typed cast).
  - `buildInsertQuery` → Firebird has **no multi-row `VALUES (…),(…)`**. A batch
    insert is rewritten to `INSERT … SELECT … UNION ALL SELECT …`, with each
    bound value wrapped in `CAST(? AS <coltype>)` (a bare `?` in a SELECT list
    has no type context → Firebird -804). Single-row inserts keep native VALUES
    (the only shape that supports RETURNING). Batches needing column defaults, or
    combining RETURNING with multiple rows, fall back to the pg builder.
- **session.ts** — FirebirdPreparedQuery/Session/Transaction over core's
  `Attachment`/`Transaction`; `run(sql, params, {rowMode:'array'})` feeds a
  locally-replicated `mapResultRow`. Nested tx (savepoints) throws for now.
- **columns.ts** — Firebird timestamp/date/time/blob/blobText with identity value
  mapping (core round-trips Date/Buffer natively). Temporal builders are named
  module-scope classes carrying the SQL type as a runtime field — an anonymous
  exported class with inherited private members can't be emitted to .d.ts (TS4094).
- **driver.ts / index.ts** — `drizzle(attachment, config)`; re-export `pgTable as
  firebirdTable` and the safe pg column types.

30 integration tests (CRUD, where eq/and/gte, FIRST/SKIP pagination, UPDATE +
single-row INSERT RETURNING, DELETE, tx commit/rollback, and a full column-type
round-trip incl. numeric/bigint/temporal/blob) green on FB3/4/5.

### The bug the drizzle work flushed out (core, not drizzle)
The column-types suite **hung**. Bisected to: a **string parameter bound to a
NUMERIC column deadlocks the connection**. Isolation (fresh connection, single
statement, wire hexdump + encodeParams instrumentation) showed:

1. **Functional**: `prepareParams` built a text codec for *any* string param via
   `codecForDesc`, which reads `subType & 0xff` as a charset id. For text columns
   that's right; for NUMERIC, `subType` is the numeric subtype (1) → resolved to
   the **OCTETS** codec, whose `encode()` throws "Cannot encode string into OCTETS
   column". (string→INTEGER/DOUBLE/VARCHAR happened to work because their subType
   is 0 → NONE codec, which encodes fine.)
2. **Robustness (the actual hang)**: that throw fired **after** the `op_execute`
   header was already written to the shared writer. The error-path teardown
   (op_free_statement + op_rollback) was then appended and flushed as one
   malformed packet; Firebird read a partial op_execute and blocked, so core's
   response read never returned → deadlock. Confirmed by decoding the 32-byte TX:
   `op_execute(63)` header only, then `op_free_statement(67)` + `op_rollback(31)`.

Fixes:
- **session.ts** — only attach a text codec to genuine text-type params
  (`isTextType(d.type)`); other string params go out as plain UTF-8 bytes and
  Firebird coerces them (`'12.5'` → NUMERIC). 
- **statement.ts** — `encodeParams` now runs **before** any bytes hit the writer,
  so an encode failure surfaces cleanly and can never desync the wire.

9 new core regression tests in `param-coercion.test.ts` (string→numeric/int/
double coercion; CAST-to-NUMERIC doesn't hang; oversized-param throws cleanly and
the connection stays alive). Core: 318 green (309 prior + 9), no regressions.

**M6 Drizzle: done.** Next M6: node-firebird compat layer + migration guide, CI,
benchmark expansion.

  node-firebird2's promises layer). Auto-upgrade swaps only the current tx, not
  the connection's base isolation.
- Pool factory + manager (`firebird2/*`, root `firebird-pool-factory`, `options`,
  `firebird-v2`, `types`) rewired to `connectAttach`; `manager.setup()` is a no-op.
- `node-firebird2` removed from deps; `@fast-firebird/core` added (file link).
  `moment`/`rxjs` added to fix a *pre-existing* build break in legacy files.
- `MIGRATION.md` documents every behavioral delta (blobs eager, `attachEvent(names)`,
  local `Isolation`, charset, `setup()` no-op, auth/wire, deps/deployment).

### Result
Full `tsc` build green. 9/9 integration tests pass on **FB3, FB4, and FB5**
(DDL/DML, params, RO→RW auto-upgrade, tx commit/rollback, `asObject` arrays,
text+binary blobs, context activation, `setIsolation`, generators, `readQuery`),
including `é`/`€` under CHARSET NONE. Existing unit spec unaffected.

### Investigated the transcode-adapter `-303` — NOT a core bug (test artifact)
Initially looked like a core defect: a win1252 `transcodeAdapter.toDb` param
raised `-303 "Malformed string"` while `charsetNoneEncoding` with the "same"
value worked. Wire-byte dump + a charset matrix settled it:
- The failing test database was created **UTF8-default** (called `createDatabase`
  without `encoding`, so `set_db_charset = UTF8`). Columns defaulted to UTF8, so
  params are UTF8-typed. The win1252 adapter dutifully sent win1252 bytes
  (`e9`/`80`) into a UTF8 param → invalid UTF-8 → `-303`. `charsetNoneEncoding`
  respected the declared UTF8 charset and sent UTF-8 (`c3a9`/`e282ac`) → OK.
- Against a real **CHARSET NONE** database (the legacy in-house convention) **both** the
  win1252 adapter and `charsetNoneEncoding` round-trip `é`/`€` correctly.

Conclusion: the adapter behaves correctly-as-designed (absolute override, like
node-firebird2). `charsetNoneEncoding` is strictly **more robust** — it respects a
real declared charset (UTF8) and only falls back to win1252 for NONE — so routing
the ext layer through it is the right call and improves on node-firebird2's
blanket adapter. No core change made.

### Demo bugs fixed (found via headless-Chrome testing)
1. **Row streaming** used a recursive CTE → Firebird's ~1024 recursion cap

  live-captured vectors (incl. 34-nines and decimal64); wired into msgcodec
  DEC16/DEC34. 57 core unit tests green. Demo now shows `1/3 →
