### Drizzle depth (#9)

- **Savepoints**: core `tx.transaction(fn)` — SAVEPOINT FF_SP_<depth> /
  RELEASE on success / ROLLBACK TO on error, names reuse per depth (cache-
  neutral; savepoint stmts aren't cacheable anyway), depth resets on
  restart(). fn receives the SAME Transaction (one server tx). Drizzle's
  FirebirdTransaction.transaction() now nests on coreTx.transaction —
  PgTransaction's dialect/session fields aren't accessible on the subclass,
  so they're captured as own constructor fields. 18 core + 12 drizzle tests
  (partial rollback, 3-level nesting, sequential scope name reuse, outer
  rollback discards released savepoints, failed-statement recovery,
  drizzle inner.rollback()).
- **Migrator**: `migrate(orm|att, {migrationsFolder, migrationsTable?})` —
  .sql files in name order, FF_MIGRATIONS tracking, full isql syntax via
  executeScript. LEARNED THE HARD WAY: perScript (one tx per file) fails —
  Firebird DML cannot see a table created in the same uncommitted tx
  ("Table unknown" on the seed insert right after create table; the reason
  isql defaults to AUTODDL). Switched to perStatement commits and documented
  the honest consequence: a failing file can be partially applied and stays
  unrecorded — keep migrations small/idempotent.
- **Introspection**: `introspectDatabase` (RDB$RELATION_FIELDS ⋈ RDB$FIELDS ⋈
  RDB$RELATIONS + PK via RDB$RELATION_CONSTRAINTS/INDEX_SEGMENTS) →
  structured model; `generateDrizzleSchema` emits firebirdTable code (camelCase
  props, NUMERIC/DECIMAL prec+scale, blob subtypes → blob/blobText, composite
  PK via primaryKey() extraConfig — now re-exported). TZ/DECFLOAT columns map
  to nearest builder with a marker comment.
- **Relational API — boundary decided by experiment**: flat findMany/findFirst
  compile to plain selects and WORK (tested FB3/4/5, incl FIRST/SKIP). Nested
  `with:` generates pg JSON-aggregation SQL → server "Token unknown"; now
  intercepted in FirebirdDialect.buildRelationalQueryWithoutPK with a clear
  "use explicit joins" error. Test gotcha: drizzle's QueryPromise.then throws
  SYNCHRONOUSLY while building SQL — wrap in a promise chain to assert as a
  rejection.

### gbak services (#10)

`svc.backup(db, fbk)` + `svc.restore(fbk, db, {replace?, pageSize?})` —
server-side paths, always-verbose (draining the log is also the completion
signal), restore creates by default / replace=0x1000 overwrites (constants
verified in consts_pub.h). New ParamBuffer.rawInt32 for SpbStart IntSpb
(tag + raw LE int32, NO length byte — unlike DPB int32). 6 tests: full
backup→restore→connect→query round trip (€ survives), replace-refusal.

### DECFLOAT specials (#11)

Decode already returned 'Infinity'/'-Infinity'/'NaN' (combo ≥ 0b11110) — the
gap was ENCODE: parseDecimal rejected them. encodeDecFloat now emits
canonical specials (sign + combo bits, zero payload), so string AND JS
number (Infinity/NaN) params bind. Server caveat surfaced by tests: equality
comparisons touching NaN rows raise Firebird's own DECFLOAT trap ("Decimal
float invalid operation") — ORDER BY total order (…< +Inf < NaN) is trap-free
and used for the sanity assertion. 8 tests (FB4/5).

### Session 37 addendum — smaller learnings worth keeping

- **False signal in the first RQB experiment**: `with:` initially failed with
  "Cannot read properties of undefined (reading 'referencedTable')" — that
  was MY schema missing `relations()` definitions, not Firebird. Only after
  defining relations did the query reach SQL generation and hit the real
  boundary (JSON aggregation). Lesson: when probing a support boundary, make
  sure the failure comes from the layer you think it does.
- **Why an own migrator instead of drizzle-kit**: drizzle-kit's dialect set
  is closed (pg/mysql/sqlite baked in) — a third-party Firebird dialect
  can't be plugged in. A plain-SQL migrator on top of our isql-faithful
  script engine is the pragmatic, honest replacement; schema DIFFING stays
  out of scope.
- **PgTransaction subclassing**: drizzle's `dialect`/`session` fields are
  TS-private on the base — a subclass can't reach them for re-instantiation.
  Captured as own constructor fields (fbDialect/fbSession/…) instead of
  fishing via casts.
- **Savepoint statements are cache-neutral by design**: StmtType.savepoint
  isn't in StatementCache.isCacheable, so SAVEPOINT/RELEASE/ROLLBACK TO
  handles are freed after each run — no LRU pollution regardless of naming;
  the depth-based FF_SP_n names are just deterministic hygiene.
- **gbak backup needs no exclusive access** — it ran fine on a database that
  had just been detached and equally works on live ones (that's gbak's whole
  point); only restore-over-existing is guarded (replace flag).
- **Services SPB dialects differ per phase**: attach-SPB uses DPB-style
  clusters (1-byte lengths, doubled version header), SpbStart uses 2-byte
  string lengths AND raw 4-byte ints with NO length byte (IntSpb). Mixing
  them up fails as "invalid parameter block" — hence the separate
  ParamBuffer.string2/rawInt32 writers.
