# 2026-07-14

## Session 35 — first CI run: red, and rightly so

Pushed CI + TZ types + backlog #7/#8 (three commits). The inaugural GitHub
Actions run failed 3 tests — `statement-cache.test.ts` ›
"statementCacheSize: 0 disables caching but everything still works" ×
FB3/4/5: `Table unknown T_CACHE_n`.

Root cause: that one test opened its zero-cache connection with
`{...FB_BASE, port}` — the SHARED test.fdb — while the suite's table lives
in the suite's own freshDb. It passed locally for months because the local
docker volumes' shared test.fdb still contained leftover T_CACHE_n tables
from BEFORE the freshDb-per-suite refactor. CI's pristine containers had no
leftovers and exposed it. Verified the diagnosis by querying rdb$relations
on all three local shared dbs (leftovers present on every one), then dropped
them so local state matches CI honesty going forward.

Fix: the secondary connection now targets the suite's database
(`database: (db as any).options.database`), like every other multi-connection
suite. Swept the rest: all other `{...FB_BASE, port}` secondary connections
only touch rdb$database, self-create their tables, or are connect-only —
consistent with CI failing exactly these 3.

Lesson recorded: the first value CI delivered was catching stale-local-state
false greens — exactly the "invisible on a dev machine" class the clean-clone
dry run was designed for, one layer deeper (database state, not filesystem).

Suite: 752 green against leftover-free local containers.
