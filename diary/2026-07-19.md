# 2026-07-19 — Session 42: Node LTS floor + node-firebird 2.11–2.14 survey

## Node.js baseline → 22

Node 20 reached end-of-life 2026-04-30, so the published `engines.node
">=20"` floor pointed at an unsupported runtime. New policy: the floor is
the **oldest supported LTS** — `>=22` (maintenance LTS until 2027-04) —
while CI and the publish workflow run **24** (active LTS until 2028-04).
Local dev stays on 22.16, so both ends of the supported range get exercised
without a CI matrix (the Firebird 4-server matrix is the expensive axis;
doubling it per Node line isn't worth it). Touched: three `package.json`
engines fields (root was still `>=18` from scaffolding), both workflow
`node-version` pins, README badge + positioning line, CONTRIBUTING. No code
changes — nothing in the driver uses post-20 APIs yet, this is a support
statement, not a feature dependency. Engines narrowing is consumer-facing,
so it ships with the next release rather than as a repo-only tweak.

## node-firebird 2.11.0–2.14.0 (all released Jul 18)

The 2.x sprint continues — four releases in one day. Cross-checked each
feature against our surface before reacting:

- **2.11 nestTables** (JOIN column collisions): we key object rows by
  alias-or-field-name, so duplicates overwrite too — but `rowMode:'array'`
  + `ColumnInfo.relation` already give a correct escape hatch, and Drizzle
  aliases every column. Convenience gap only → backlog #19, demand-driven.
- **2.12 savepoints / metadata / tagged templates**: savepoints shipped
  here in the backlog-close session (`tx.transaction()` = depth-named
  SAVEPOINT nesting, restart-safe); column metadata is on every result;
  tagged-template SQL is Drizzle's job.
- **2.13 single-byte codepages + batchStream + ESM**: our charset table has
  carried WIN1250–1258, ISO8859_2–13, KOI8R/U and the DOS pages since the
  M0 charset layer (the CHARSET NONE/win1252 requirement forced the whole
  family early);
  ESM+CJS conditional exports since packaging. **batchStream is the real
  find**: a bulk-insert stream. Theirs is client-side batching; the
  protocol-level answer is the FB4+/P16 wire batch API
  (op_batch_create/msg/exec) — many rows per round trip, squarely our
  thesis. Backlog #17, top of the post-0.1.0 feature list.
- **2.14 PoolCluster** (multi-host primary/replica failover): we have a
  single-host pool with validation-on-borrow; a cluster is a routing/health
  layer on top. Backlog #18, demand-driven.

Net: two genuine gaps (wire batch, multi-host pool), both queued; the rest
was already covered — worth the hour of verification before assuming the
release notes described gaps.
