## What shipped
- **New pairwise scenario: `scan 300×8KB blobs`** — the "table of memos /
  documents" shape where per-blob round trips dominate. 8KB fits FB5's inline
  limit (≤64KB), so fast-firebird's *default* config fetches all 300 blobs
  with zero blob round trips (21 RTs total = prepare/execute/fetch batches),
  while node-firebird opens/reads/closes each blob sequentially via its
  callback API. Measured: **27× (0ms) / 123× (2ms) / 133× (10ms one-way)**.
- **Blob strategy matrix** (fast-firebird only, `BENCH_BLOB=1` to run alone):
  the same scan under eager+inline (default), eager with `maxInlineBlobSize:0`
  (pipelined open/read/close), and `queryStream` lazy ± `blobReadAhead` ±
  simulated per-row consumer work. Round trips reported per cell.
- **Full re-measure of the pairwise suite** (node-firebird 1.1.10, FB5,
  Node 22.16): 1MB blob ×3 is now 80 RT (was 131) and 360ms @2ms (was 950) —
  the July 10/11 pipelining work paying off — pushing the blob speedup to
  **11× / 109× / 149×**. Non-blob scenarios essentially unchanged (selects
  2.6–3.0×, inserts 1.6–2.0×, scans 1.1–1.6×).
- README performance section rewritten: refreshed pairwise table (now with a

  from" strategy table with honest interpretation. `plans/performance.md` got

## The readAhead "anomaly" — a lesson in benchmarking what a feature is FOR
First matrix run: `blobReadAhead` showed **607 RTs and the same wall time as
on-demand** (3874 vs 3916 ms @2ms). Looked like a broken feature. It isn't —
reading `BlobPrefetcher` (api/blob-prefetch.ts) explains it: the prefetcher is
deliberately single-flight, fetching one blob at a time in bounded op-lock
slices so consumer operations can interleave. It never reduces round trips;
it *overlaps* them with the consumer's per-row processing time. My scan loop
did `await blob.buffer()` and nothing else — zero work to hide latency behind,
so read-ahead correctly degenerated to on-demand.

Fix: benchmark the designed use case. Added ±10ms/row simulated consumer work
(a disk write / image resize stand-in). Result @2ms delay: on-demand 7546 ms
vs readAhead **3944 ms ≈ the 3847 ms pure-fetch floor** — the consumer's 3s of
work is fully hidden (1.9×). At 10ms delay the sequential fetch dominates
(14690 vs 15038 pure fetch): readAhead's ceiling is the single-flight fetch
pipeline, which is the documented trade-off (bounded memory, interleaving).

Two takeaways worth keeping:
1. A feature benchmarked outside its design envelope will look broken.
   Verify WHAT a mechanism trades before concluding it fails (same lesson
   shape as the RQB false signal on 07-15).
2. The strategy matrix is more persuasive than a single speedup number
   because it shows *where* the win comes from: inline = 21 RT, pipelined
   eager = 217 RT (~0.7 RT/blob), lazy = 607 RT (2/blob) — and readAhead as
   overlap, not RT reduction. Published exactly that framing in the README.
