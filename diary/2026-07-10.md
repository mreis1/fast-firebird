### Timestamp / timezone semantics (from the 07-09 feature-card question)
- `TIMESTAMP/TIME WITH TIME ZONE` decode to **JS `Date`s = pure UTC instants**.
  The wire carries UTC + a zone id; core keeps the instant and discards the
  zone id by design. Verified correct: `09:30 America/New_York (EDT, UTC−4)` →
  `2024-06-15T13:30:00.000Z`.
- Consequence: `X` and `X AT TIME ZONE 'UTC'` are the *same instant* and render
  identically once in a JS Date — not a bug, but invisible in a demo. The demo
  card now renders zoned values as varchar server-side (09:30 America/New_York ·
  13:30 UTC · 15:30 Europe/Paris) next to the raw instant.
- Plain `TIMESTAMP`/`TIME`/`DATE` (no zone) decode as **local wall-clock**
  Dates (constructed via local-time components) — matching how zoneless
  Firebird values are conventionally used.
- If an app must round-trip the zone itself, that's backlog item 2
  (`{date, zone}` value or Temporal ZonedDateTime once stable in Node).

## Session 16 — Legacy_Auth server-initiated fallback (~08:15)

**User hit on a real remote server** (`AuthServer = Srp, Legacy_Auth`,
`UserManager = Legacy_UserManager, Srp`, `WireCrypt = Enabled`):
`Server switched to unsupported auth plugin 'Legacy_Auth'`.

**Root cause chain** (reproduced in a scratch container mirroring that conf,
`fast-firebird-test-srponly` on :30507, auto-removed after):
- Account exists in BOTH managers but the **passwords drifted** — the SRP
  verifier holds a stale password (classic gsec-managed server; gsec only
  writes the legacy security db), the legacy one is current.
- Server: Srp proof fails → falls through to Legacy_Auth mid-handshake.
  fbclient follows the switch (its default `AuthClient = Srp256, Srp,
  Legacy_Auth`); we threw at the switch. With synced passwords the topology
  worked fine all along — plain-Srp restart path (server without Srp256) was
  verified correct.

**Fix (packages/core/src/protocol/handshake.ts + constants.ts):**
- `AUTH_PLUGIN_LIST` → `Srp256,Srp,Legacy_Auth` (matches fbclient default).
- Handshake follows a server switch to Legacy_Auth in every phase: at accept,
  during the empty-data restart rounds, mid-proof (cont_auth after a rejected
  SRP proof — sends the DES token, clears the unconfirmed session key), and at
  attach time in `readResponseWithAuth` (computes the token from pending
  credentials). Legacy is one-shot: any re-offer after the token = proper
  login error (no hang, no infinite resend — replaced the old unconditional
  resend in readResponseWithAuth).
- SRP→SRP plugin switch with empty data now re-offers the public key instead
  of silently ignoring it.
- `wireCrypt: 'required'` + legacy fallback → explicit error (legacy yields no
  session key, so no wire encryption is possible). Server `Enabled` + default
  client setting degrade gracefully to plaintext.
- Extracted `drainLegacyResponse` + `LOGIN_ERROR` helpers.

**Verified:** drifted account → Legacy_Auth; legacy-only account without
`authPlugin` → Legacy_Auth; synced account still prefers Srp+Arc4; wrong
password everywhere → login error in ms. 4 new integration tests on fblegacy
(server-initiated fallback describe block). **356 tests green, 2× stable.**

## Session 17 — WAN blob pipelining, DDL lock-conflict root causes, demo lifecycle (~11:30)

User field report (remote DB over fiber, demo custom bench, 125×1.8 MB blobs):
60 s insert, 100 s fetch (~2 MB/s), then `lock conflict on no wait transaction …
object TABLE "FF_CUSTOM_BENCH" is in use` on re-run; IBExpert later showed
"table id 654 is not defined" (that one is IBExpert's stale metadata snapshot —
its tx predates our recreate; commit/reconnect there fixes it, not a driver bug).

### 1. Blob segment I/O was serial → pipelined (core)
- `writeBlob`: flush+await per 64 KB op_batch_segments; `readBlob`: one
  round trip per op_get_segment. A 1.8 MB blob ≈ 30 RTs → exactly the user's
  timings at WAN RTT.
- Wire-probed post-EOF behaviour: op_get_segment after EOF returns a CLEAN
  empty EOF response (FB3/4/5) — safe to over-read. So both paths now run a
  32-deep window (`PIPELINE_DEPTH`, ~2 MiB in flight), errors surface in
  order, error paths drain in-flight responses so the connection stays usable.
- A/B through a 15 ms/dir latency proxy (5×1.8 MB): insert 5972→890 ms,
  fetch 5490→888 ms (~6.5×; proxy itself became the bandwidth cap).
  User's 60 s/100 s should drop to roughly 10-15 s each.

### 2. TWO core bugs behind the "in use" / DDL failures
- **Statement-cache metadata pins**: cached prepared statements pin table
  metadata server-side; the demo pool's caches held bench statements →
  recreate failed. New API: `Attachment.clearStatementCache()` (clears +
  flushes the deferred frees via one ping) and `Pool.clearStatementCaches()`
  (idle conns flushed in parallel; in-use cleared lazily).
- **Commit-failure transaction leak** (the nasty one): Firebird takes DDL
  metadata locks at COMMIT time, so "object in use" surfaces on op_commit.
  `Transaction.commit()` marked `finished = true` BEFORE awaiting — a refused
  commit left the tx marked finished client-side but ALIVE server-side →
  `attachment.transaction()` skipped rollback → leaked active tx → every later
  DDL failed with "update conflicts with concurrent update, concurrent
  transaction N" (deterministic, 5/5 rounds). Fix: commit/rollback revert
  `finished` on failure so the owner's rollback actually runs. (MON$ red
  herring en route: Garbage Collector/Cache Writer system attachments hold
  idle transactions — filter MON$SYS before blaming a leak.)
- Demo bench now: `clearStatementCaches()` before DDL + `wait: 10` default
  (nowait immediately after a failed-commit rollback races server cleanup).

### 3. Demo enhancements (user requests)
- Explicit connect/disconnect per server: `disconnectServer()` keeps config,
  closes pool/drizzle/compat; `connected` flag in /api/servers; UI disconnect
  button on the Connection panel + gated "disconnected" card with Connect.
- `TxWaitPicker` (default | wait | no wait | wait N s → TPB wait/nowait/
  lock_timeout, already supported by core): SQL runner (core lane; note shown
  for drizzle/compat) and custom-bench DDL (default label "wait 10 s").
- Micro-benchmark recreate got the same clear+wait treatment.

### Verified
- Tests: 12/12 pool (2 new: DDL-pin eviction, commit-leak regression),

- Browser (headless Chrome): custom bench twice in a row (no lock conflict,

  note, blob round-trip (café € intact), zero console errors.

## Session 19 — wire compression: how it works + demo toggle (~12:45)

User question: "compression is always off — how is it enabled?"

### How Firebird wire compression is negotiated (the important stuff)
- **Both sides must opt in.** It is decided ONCE, at the protocol handshake:
  the client sets the compression flag (`pflag_compress`) on its protocol
  offers in `op_connect`; the server accepts it only when `WireCompression =
  true` in firebird.conf (accept packet echoes the flag). There is no way to
  turn it on later — changing it means reconnecting.
- **Client default is OFF** — in fbclient AND in fast-firebird
  (`wireCompression: false` in `resolveOptions`). That's why every demo
  connection showed "off" even though the docker matrix servers have
  `FIREBIRD_CONF_WireCompression: "true"`: the demo simply never requested it.
  Server conf alone does nothing; client option alone does nothing.
- Once negotiated, zlib covers **every byte after the accept packet, both
  directions** (deflate stream, not per-packet), and it **composes with wire
  encryption**: compress → encrypt on send (compressing ciphertext would be
  useless). Verified live: FB5 connection showing encryption Arc4 +
  compression zlib simultaneously.
- **When it helps**: text-ish payloads (rows, varchar, XML/JSON-in-blob,
  RDB$ metadata) over slow links — often several×. **When it hurts**: already
  compressed blobs (JPG/PDF/ZIP — the user's exact payload) burn CPU for ~0%
  gain; on a LAN it's usually a net loss. So client-side default-off is the
  right call; enable per-connection where the link is thin and data is
  compressible.
- fbclient equivalent: `WireCompression = true` in the CLIENT's firebird.conf
  (or isc_dpb_config); fast-firebird: `connect({ wireCompression: true })`.

### Demo: compression is now first-class
- `ServerConfig.wireCompression` + single `connectOptsFor()` builder used by
  EVERY lane (pool, drizzle attachment, compat manager, events SSE, stream
  SSE — the last two were building their own opts and would have silently
  diverged).
- `POST /api/servers/:id/config` patches the flag and tears down live state
  (handshake-time setting → applies on reconnect).
- UI: "zlib compression" checkbox on the add-server form; the Connection

  toggling remounts the panels and the badge flips off↔zlib live (verified
