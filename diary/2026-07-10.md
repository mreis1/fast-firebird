### Timestamp / timezone semantics (from the 07-09 feature-card question)
- `TIMESTAMP/TIME WITH TIME ZONE` decode to **JS `Date`s = pure UTC instants**.
  The wire carries UTC + a zone id; core keeps the instant and discards the
  zone id by design. Verified correct: `09:30 America/New_York (EDT, UTC−4)` →
  `2024-06-15T13:30:00.000Z`.
- Consequence: `X` and `X AT TIME ZONE 'UTC'` are the *same instant* and render
  identically once in a JS Date — not a bug, but invisible in a demo. The demo
  card now renders zoned values as varchar server-side (09:30 America/New_York ·
  13:30 UTC · 15:30 Europe/Paris) next to the raw instant.
- Plain `TIMESTAMP`/`TIME`/`DATE` (no zone) decode as **local wall-clock**
  Dates (constructed via local-time components) — matching how zoneless
  Firebird values are conventionally used.
- If an app must round-trip the zone itself, that's backlog item 2
  (`{date, zone}` value or Temporal ZonedDateTime once stable in Node).

## Session 16 — Legacy_Auth server-initiated fallback (~08:15)

**User hit on a real remote server** (`AuthServer = Srp, Legacy_Auth`,
`UserManager = Legacy_UserManager, Srp`, `WireCrypt = Enabled`):
`Server switched to unsupported auth plugin 'Legacy_Auth'`.

**Root cause chain** (reproduced in a scratch container mirroring that conf,
`fast-firebird-test-srponly` on :30507, auto-removed after):
- Account exists in BOTH managers but the **passwords drifted** — the SRP
  verifier holds a stale password (classic gsec-managed server; gsec only
  writes the legacy security db), the legacy one is current.
- Server: Srp proof fails → falls through to Legacy_Auth mid-handshake.
  fbclient follows the switch (its default `AuthClient = Srp256, Srp,
  Legacy_Auth`); we threw at the switch. With synced passwords the topology
  worked fine all along — plain-Srp restart path (server without Srp256) was
  verified correct.

**Fix (packages/core/src/protocol/handshake.ts + constants.ts):**
- `AUTH_PLUGIN_LIST` → `Srp256,Srp,Legacy_Auth` (matches fbclient default).
- Handshake follows a server switch to Legacy_Auth in every phase: at accept,
  during the empty-data restart rounds, mid-proof (cont_auth after a rejected
  SRP proof — sends the DES token, clears the unconfirmed session key), and at
  attach time in `readResponseWithAuth` (computes the token from pending
  credentials). Legacy is one-shot: any re-offer after the token = proper
  login error (no hang, no infinite resend — replaced the old unconditional
  resend in readResponseWithAuth).
- SRP→SRP plugin switch with empty data now re-offers the public key instead
  of silently ignoring it.
- `wireCrypt: 'required'` + legacy fallback → explicit error (legacy yields no
  session key, so no wire encryption is possible). Server `Enabled` + default
  client setting degrade gracefully to plaintext.
- Extracted `drainLegacyResponse` + `LOGIN_ERROR` helpers.

**Verified:** drifted account → Legacy_Auth; legacy-only account without
`authPlugin` → Legacy_Auth; synced account still prefers Srp+Arc4; wrong
password everywhere → login error in ms. 4 new integration tests on fblegacy
(server-initiated fallback describe block). **356 tests green, 2× stable.**

**For the user's remote server (optional server-side fix):** re-sync with
`ALTER USER <name> PASSWORD '<pw>' USING PLUGIN Srp` to get SRP + wire
encryption back for that account.
